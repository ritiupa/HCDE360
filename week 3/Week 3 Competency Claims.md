## C3 -- Data Cleaning and File Handling

The messy CSV had several real problems: R005 had no name or role, R009 had "fifteen" written out as a word in the `experience_years` column, and roles were inconsistently cased ("ux designer", "UX DESIGNER", "UX Designer" all appeared). The script handles all of these: rows with missing names or roles are skipped before they reach any calculation, `parse_experience_years` converts word-form numbers using a lookup dictionary and returns None for anything unrecognizable, and `.title()` normalizes casing so role counts don't split across variants. I found two bugs in the terminal and three through Cursor agent. I asked for the fixes and their reasoning through Cursor's agent. Bug 1 was a ValueError from trying to parse "fifteen" as an integer directly; Bug 2 was a sort direction error that returned the lowest satisfaction scores instead of the highest; Bug 3 was a missing blank-check on role before counting; Bug 4 was including rows with empty participant names in the top 5 list; Bug 5 was casting satisfaction_score to int without first checking it was numeric.

## What would happen if you ran this script on a real dataset instead of the demo CSV?

On a real dataset, the first thing likely to break is the `experience_years` column. The `word_to_number` dictionary only covers integers zero through twenty -- anyone with 21 or more years of experience who typed it as a word ("twenty-two", "thirty") would return None and get silently dropped from the analysis. You would know something was wrong only if you compared the number of rows printed in "Responses by role" against the total row count in the file and noticed the numbers didn't match. A better approach would be to log every skipped row with its ID and the reason it was skipped, rather than dropping it silently.

A second risk is the `.title()` normalization on role. It converts "ux researcher" to "Ux Researcher" correctly, but it also converts "UX Researcher" to "Ux Researcher" -- which may not match how roles are labeled in other systems. If this output was joined to another dataset where role was stored as "UX Researcher", the capitalization mismatch would cause rows to fail to match. You would not notice this at all from the script's terminal output -- it would only surface when comparing results downstream.
